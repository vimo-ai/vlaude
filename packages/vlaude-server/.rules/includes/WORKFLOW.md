### 开发工作流（Dev & PR Workflow）

#### 日常开发步骤
1) 明确需求与边界：记录预期行为、非目标与影响范围
2) 发现根目录：识别 `<backend_root>` 与 `<frontend_root>`
3) 设计变更：
   - 涉及的上下文/模块与层（api/application/domain/infrastructure）
   - 领域对象/不变式影响；是否需要新聚合/值对象
   - 可选模块（ORM/权限/对象存储/Outbox）是否存在与启用
4) 列出将读取/修改的文件清单与原因
5) 实施顺序：
   - 先接口/领域，再实现/编排，再 I/O/适配，再 API/视图
   - 后端：接口→领域→应用→基础设施→控制器
   - 前端：types→services/api→stores/composables→views/components→router
6) 自测：typecheck → lint → unit → integration → e2e（按需），一次性运行
7) 提交：小步提交，解释“为何”，引用 Issue；开启 PR 并附测试计划与回滚方案

#### 代码评审清单（Reviewer）
- 职责边界是否清晰？是否违反分层/依赖方向？
- 是否引入了未启用模块的死引用？
- 是否正确使用上下文对象传递身份/租户？
- 是否具备足够测试，能否复现与覆盖风险？
- 是否存在安全/性能风险或兼容性问题？

#### CI 基线
- 安装依赖（遵循仓库包管理器）
- 类型检查、Lint、测试（非 watch）
- 覆盖率报告与失败中断

#### 发布与回滚（可选）
- SemVer 与变更日志
- 数据迁移（若启用 ORM）：备份、幂等回滚脚本
- 灰度/开关：可控启用与快速回退
